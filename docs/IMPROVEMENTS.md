# แผนพัฒนาระบบ TB Medication Tracker

เอกสารนี้สรุปผลการตรวจสอบระบบ (มิถุนายน 2026) แบ่งเป็นสิ่งที่ดำเนินการแล้ว
และแผนพัฒนาในอนาคต

---

## 1. ดำเนินการแล้วในรอบนี้

### การแก้ไขระบบ (System Fixes)

| รายการ | เหตุผล |
|---|---|
| บังคับตั้ง `SECRET_KEY` ใน production | เดิมมีค่า default อ่อนแอ (`change_this_secret`) ถ้าลืมตั้ง env var จะปลอม session/cookie ได้ — ตอนนี้แอป fail fast ทันทีถ้าไม่ตั้ง |
| เพิ่ม session cookie flags | `HTTPONLY` + `SAMESITE=Lax` ทุก config และ `SECURE` ใน production ป้องกัน XSS ขโมย cookie และ CSRF |
| ยืนยันกินยาแบบ atomic | เดิมเช็คแล้วค่อย commit (race condition) — เปลี่ยนเป็น conditional UPDATE `WHERE taken=false` กดพร้อมกันหลายครั้งจะไม่ทับเวลาเดิม |
| Rate limiting ที่ `/scan/<token>` | endpoint สาธารณะ เสี่ยงโดน brute-force token — จำกัด 30 คำขอ/นาที/IP |
| Audit log ไม่ล้มแบบเงียบ | เดิมใช้ `print()` — เปลี่ยนเป็น `app.logger.exception` + rollback session ที่พังจาก commit ล้มเหลว |
| `/ping` เช็ค database จริง | health check ทำ `SELECT 1` คืน 503 เมื่อ DB ล่ม ให้ระบบ monitor ตรวจจับได้ |
| Validate year/month query params | เดิม `?month=13` ทำให้หน้า report/ปฏิทิน/print เกิด 500 — ตอนนี้ fallback เป็นเดือนปัจจุบัน |
| Whitelist ชื่อยา | ฟอร์มกรอกยาทุกจุดรับเฉพาะยาที่อยู่ใน `DRUG_IMAGES` ป้องกันข้อมูลยาผิดเข้าระบบ |
| ตั้งค่า gunicorn ชัดเจน | กำหนด workers/threads/timeout และ access log ใน Procfile |

### ฟีเจอร์ใหม่

**จัดการบัญชีเจ้าหน้าที่ (`/staff`)**
- บัญชีเก็บใน database พร้อมบทบาท: ผู้ดูแลระบบ / เภสัชกร / พยาบาล
- ผู้ดูแลระบบเพิ่ม/แก้ไข/ปิดใช้งานบัญชี และรีเซ็ตรหัสผ่านได้โดยไม่ต้อง deploy ใหม่
- ทุกคนเปลี่ยนรหัสผ่านตัวเองได้ที่ `/staff/password`
- บัญชีจาก env vars (`STAFF_USER`/`STAFF_PASS_HASH`) ยังใช้ได้เป็น bootstrap
  (ได้สิทธิ์ผู้ดูแลระบบ) — แนะนำให้สร้างบัญชีใน DB แล้วลบ env vars ภายหลัง

**ผู้ป่วยแจ้งอาการไม่พึงประสงค์ (หน้าสแกน QR)**
- เลือกอาการจาก 7 หมวด (คลื่นไส้, ผื่นคัน, ตัวเหลือง-ตาเหลือง, ชาปลายมือเท้า,
  ตามัว, ปวดข้อ, อื่นๆ) + ข้อความเพิ่มเติม
- ระบบตอบคำแนะนำเบื้องต้นทันที — อาการรุนแรง (ตัวเหลือง, ตามัว) แสดงคำเตือน
  "หยุดยาและติดต่อคลินิกทันที"
- เภสัชกร/ผู้ดูแลระบบตอบกลับได้ที่หน้า `/symptoms` ผู้ป่วยเห็นคำตอบเมื่อสแกน QR อีกครั้ง
- Badge ใน navbar แจ้งจำนวนรายการใหม่ที่ยังไม่ได้ตอบ
- จำกัด 3 รายงาน/วัน/ผู้ป่วย ป้องกัน spam

**LINE แจ้งเตือนอาการไปหาเภสัชกร (optional)**
- เมื่อตั้งค่า LINE Messaging API ครบ ระบบจะ push การแจ้งอาการไปหาเภสัชกรที่ลงทะเบียน
  พร้อมชื่อ/HN/อาการ และ "เลขรับคำตอบ" (A01)
- เภสัชกรลงทะเบียนเองใน LINE ด้วยรหัสลับ (`LINE_REGISTER_CODE`)
- เภสัชกรตอบกลับใน LINE ด้วย `A01+ข้อความ` → ระบบนำคำตอบมาแสดงในเว็บให้ผู้ป่วย
  (ไม่แสดงเลขรับคำตอบ)
- เลขรับคำตอบถูกจองไว้จนกว่าจะปิดเรื่อง แล้วนำกลับมาใช้ใหม่ได้ — กัน reply หลงเรื่อง
- ตรวจ webhook signature (HMAC-SHA256) ทุกครั้ง ใช้ stdlib ไม่เพิ่ม dependency
- ถ้าไม่ตั้งค่า LINE ระบบทำงานปกติ ไม่ส่ง LINE และ `/line/webhook` คืน 404

### การตัดสินใจเชิงออกแบบที่จงใจคงไว้

- **`WTF_CSRF_TIME_LIMIT = None`** — CSRF token ผูกกับ session (อายุ 8 ชม.) อยู่แล้ว
  การตั้งหมดอายุสั้นกว่านั้นมีแต่ทำให้ฟอร์มที่เปิดค้างไว้นาน submit ไม่ผ่าน
- **endpoint สแกนยกเว้น CSRF** — token 43 ตัวอักษรใน URL คือตัวยืนยันตัวตนอยู่แล้ว
  ผู้โจมตีที่ปลอมคำขอได้ต้องรู้ token ซึ่งเท่ากับเข้าถึงหน้าสแกนได้โดยตรงอยู่ดี
- **Rate limiter เก็บใน memory** — นับแยกต่อ gunicorn worker (limit จริง = limit × workers)
  และรีเซ็ตเมื่อ restart ซึ่งยอมรับได้สำหรับคลินิกขนาดเล็ก
  (อนาคต: Flask-Limiter + Redis ถ้า scale ขึ้น)
- **Session เก่าที่ไม่มี role ถือเป็น admin** — รองรับ session ที่ login ไว้ก่อน deploy
  (เดิมทุกคนมีสิทธิ์เต็มอยู่แล้ว) ผลคือ permissive ได้นานสุด 8 ชม. หลัง deploy

## 2. แผนพัฒนาในอนาคต

เรียงตามลำดับความสำคัญโดยประมาณ

### ความสำคัญสูง
- **Reminder ผ่าน LINE ถึงผู้ป่วย** — ต่อยอดจาก LINE integration ที่มีแล้ว
  เตือนผู้ป่วยที่ยังไม่สแกนยืนยันในแต่ละวัน และแจ้งเจ้าหน้าที่อัตโนมัติเมื่อผู้ป่วย
  กลายเป็น at-risk (ปัจจุบันต้องเปิด dashboard เอง)
  ต้องมี background job (APScheduler/Celery) และตาราง notification กันส่งซ้ำ
- **DOT (Directly Observed Therapy)** — แยกบันทึก "ผู้ป่วยกดยืนยันเอง" กับ
  "เจ้าหน้าที่เห็นการกินจริง" (เพิ่ม `witnessed_by` ใน MedicationDose + หน้าสแกนฝั่งเจ้าหน้าที่)
- **กลยุทธ์ backup ฐานข้อมูล** — เอกสารและ automation สำหรับ pg_dump/restore
  ปัจจุบันการลบถาวร (hard delete) กู้คืนไม่ได้

### ความสำคัญกลาง
- **ตรวจสอบระยะเวลารักษาตามชนิด TB** — เตือนเมื่อสร้างตารางสั้นกว่ามาตรฐาน
  (PTB 6 เดือน, EPTB 9 เดือน) หรือปิด outcome "completed" ก่อนครบกำหนด
- **ข้อมูลคลินิกเพิ่มเติมของผู้ป่วย** — โรคร่วม (HIV, เบาหวาน, โรคไต/ตับ),
  ประวัติแพ้ยา, การตั้งครรภ์ — ต่อยอดเป็นการปรับสูตรยาอัตโนมัติและคำเตือน
- **บันทึก Adverse Event เชิงคลินิก** — ต่อยอดจาก SymptomReport ของรอบนี้
  เพิ่ม severity/causality assessment และรายงาน pharmacovigilance
- **Export ตามฟอร์มมาตรฐาน สสจ./WHO (NTIP)** — ลดการกรอกข้อมูลซ้ำเข้าระบบรายงานระดับประเทศ
- **เก็บ `taken_time` แบบ timezone-aware** — ปัจจุบันเก็บ naive datetime (เวลาไทย)
  ควรเปลี่ยนเป็น UTC ภายในและแปลงตอนแสดงผล (ต้อง migrate ข้อมูลเดิม)

### ความสำคัญต่ำ / ระยะยาว
- **Multi-clinic** — เพิ่มตาราง Clinic แยกข้อมูลผู้ป่วย/เจ้าหน้าที่ต่อหน่วยงาน
  (จำเป็นถ้าจะใช้ deployment เดียวหลายโรงพยาบาล)
- **Pharmacy inventory** — บันทึกการจ่ายยา เทียบกับการกินจริง และแจ้งสต๊อก
- **Contact tracing** — ฟอร์มสอบสวนโรคและรายชื่อผู้สัมผัส
- **Dashboard ประสิทธิภาพเจ้าหน้าที่** — สรุปกิจกรรมจาก audit log
- **Redis-backed rate limiting / login lockout** — เมื่อรันหลาย worker หรือหลาย instance
